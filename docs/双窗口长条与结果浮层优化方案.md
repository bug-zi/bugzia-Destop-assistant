# Bugzia 双窗口长条与结果浮层优化方案

> 更新时间：2026-06-19
> 目标：让 Bugzia 在常驻桌面时始终保持“搜索框长条”的美观形态，同时在需要交互时通过临时结果浮层承载 AI 回复、文件搜索结果与浏览器搜索反馈。

> **实施进度（2026-06-19，MVP 已达成）**：方案已全量落地，实际演进为**三窗口**——设置页从主窗口内嵌面板升级为独立 `settings` 窗口（功能完整、经 `settings:updated` 事件同步、不写盘）。落地情况：
> - **骨架**——`App.tsx` 按 window label 路由（main / result / settings）、三份按 label 分窗口的 capability、`WebviewWindow` 动态创建基建：完成。
> - **设置窗口**——完成（外观 / AI / 搜索 / 卡片四段，含连接测试）。
> - **结果窗口（阶段 1–5）**——完成：动态创建 / 近主窗口定位 / `result:ready` + `replay` 握手 / 流式镜像 / 固定 / Esc / 停止 / 清空（主窗口为驱动方与镜像源）；`/file` 文件搜索结果已在结果窗口由 `FileResultsView` 承载。
> - **阶段 6 设置项**——结果窗口几何（`result_x` / `result_y` / `result_w` / `result_h`）**全部持久化**，重启后恢复用户放置的位置与尺寸。
> - **主窗口**——已固化为长条（`maxHeight:72` + 启动强制 64、移除 `expanded` 拉高与内嵌 `result-panel`，仅留拖动手柄 + spark 切换按钮 + 输入框 + 一行短状态）。
>
> 下文"双窗口"的设计目标与阶段划分仍然适用，区别只是设置单独成窗、结果内容由一个 `result` 窗口承载。文件清单与逐任务实现见 [`开发进度.md`](开发进度.md)。静态闸口（`tsc --noEmit` / `cargo check` / `pnpm build`）全绿；§14 验收的运行时交互仍需用 `pnpm tauri dev` 在桌面手动验证。本文件正文为当时的**设计依据**，保留作为决策记录。

## 1. 背景问题

Bugzia 的产品定位是一个常驻桌面的半透明 AI Command Card。它首先是桌面美化组件，其次才是 AI 与搜索入口。

当前实现中，`CommandCard` 在展开态会把同一个 Tauri 主窗口从长条拉高为矩形，用 `result-panel` 承载聊天内容。这个方案实现简单，但会带来一个核心体验问题：

- 常驻态是漂亮的搜索框长条。
- 使用功能时窗口变成大矩形。
- 大矩形长期停留在桌面上会破坏桌面装饰感。

因此需要把“常驻美观入口”和“临时交互内容”从视觉形态和窗口生命周期上分离。

---

## 2. 设计目标

### 2.1 核心目标

1. 主窗口永远保持长条搜索框形态，不因 AI 回复或搜索结果而变宽、变高。
2. 结果内容由独立的临时浮层窗口承载。
3. 浮层只在需要时出现，可关闭、可自动隐藏、可固定。
4. 主窗口和浮层共享同一套设置、主题、输入状态与功能流。
5. 不改变 Bugzia “轻量、半透明、桌面美化”的产品气质。

### 2.2 非目标

第一阶段不做以下内容：

- 不做完整多窗口工作区。
- 不做复杂会话管理器。
- 不做插件化布局。
- 不做桌面嵌入 WorkerW。
- 不重构 AI SSE 后端协议。
- 不引入新的前端路由框架，除非后续确有必要。

## 3. 推荐方案总览

采用“双窗口架构”：

```text
┌──────────────────────────────┐
│ 主窗口 main                   │
│ 长条 Command Bar              │
│ Ask or search...          ⚙  │
└──────────────────────────────┘

        ┌──────────────────────────────┐
        │ 结果窗口 result              │
        │ AI 回复 / 文件结果 / 状态反馈 │
        │ 可滚动、可关闭、可固定        │
        └──────────────────────────────┘
```

### 3.1 主窗口职责

主窗口 `main` 只负责：

- 常驻桌面长条 UI。
- 输入内容。
- 设置入口。
- 拖动与锁定。
- 简短状态显示，例如“生成中”“已回复”“3 个文件结果”。
- 决定何时打开、更新、隐藏结果浮层。

主窗口不再直接承载长内容。

### 3.2 结果窗口职责

结果窗口 `result` 负责：

- AI 流式回复。
- 聊天消息列表。
- 文件搜索结果列表。
- 搜索/错误/空状态。
- 停止生成、清空上下文、关闭、固定等操作。

结果窗口是临时窗口，不进入任务栏，不抢占产品主形态。

## 4. 用户体验规则

### 4.1 状态模型

建议将体验分为 4 个状态：

| 状态 | 主窗口表现 | 结果窗口表现 | 说明 |
|---|---|---|---|
| Idle | 仅显示长条 | 隐藏 | 桌面美化常驻态 |
| Focus | 长条轻微高亮 | 隐藏或保持原状态 | 用户准备输入 |
| Active | 长条仍保持尺寸，显示短状态 | 显示并跟随主窗口定位 | AI / 文件搜索交互中 |
| Pinned | 长条仍保持尺寸 | 结果窗口保持显示 | 用户主动固定结果 |

### 4.2 推荐交互

1. 用户聚焦输入框，主窗口不展开。
2. 用户按 Enter：
   - 普通文本或 `/ai`：打开结果窗口并开始流式输出。
   - `?` 或 `/web`：打开浏览器，同时主窗口短暂显示“已打开浏览器搜索”；结果窗口可不打开。
   - `/file`：打开结果窗口并显示文件结果。
3. 用户按 Esc：
   - 若结果窗口未固定，隐藏结果窗口。
   - 主窗口保持长条。
4. 用户点击结果窗口关闭按钮：
   - 隐藏结果窗口。
   - 不清空聊天上下文。
5. 用户点击结果窗口图钉：
   - 结果窗口进入固定状态。
   - Esc 不再自动隐藏结果窗口，除非再次取消固定。

### 4.3 主窗口短状态

主窗口右侧可以显示短状态，但不能拉伸布局：

```text
Ask or search...              生成中
Ask or search...              已回复
Ask or search...              12 个结果
Ask or search...              错误
```

状态文字应有最大宽度，超出省略，避免破坏长条美感。

## 6. 窗口设计

### 6.1 主窗口 main

主窗口保持当前透明无边框配置，但尺寸固定为长条逻辑：

```json
{
  "label": "main",
  "title": "Bugzia",
  "width": 420,
  "height": 64,
  "minWidth": 360,
  "minHeight": 64,
  "decorations": false,
  "transparent": true,
  "resizable": true,
  "skipTaskbar": true,
  "shadow": false,
  "visible": false
}
```

调整原则：

- 不再用 AI 结果驱动 `main` 的高度。
- `window.expanded` 不再表示主窗口展开，而应迁移为“结果窗口是否显示”或废弃。
- 主窗口仍可允许用户调整宽度，但高度建议固定在 56 到 72 之间。

### 6.2 结果窗口 result

结果窗口建议由代码动态创建，而不是写死在 `tauri.conf.json` 中。原因：

- 可以按需创建，降低启动干扰。
- 可以根据主窗口位置动态决定浮层位置。
- 后续可支持多显示器边界修正。

推荐默认配置：

```text
label: result
width: 420
height: 360
minWidth: 360
minHeight: 220
decorations: false
transparent: true
resizable: true
skipTaskbar: true
shadow: false
visible: false
```

结果窗口默认不置顶，延续当前产品要求：它可以被其他窗口覆盖。

## 7. 结果窗口定位策略

### 7.1 默认定位

结果窗口默认出现在主窗口下方，并左对齐：

```text
result.x = main.x
result.y = main.y + main.height + 8
result.width = main.width
result.height = savedResultHeight || 360
```

### 7.2 边界修正

如果主窗口靠近屏幕底部，结果窗口应显示在主窗口上方：

```text
if main.y + main.height + gap + result.height > monitor.bottom:
  result.y = main.y - gap - result.height
```

如果主窗口靠近屏幕右侧：

```text
if main.x + result.width > monitor.right:
  result.x = monitor.right - result.width - margin
```

如果主窗口宽度太窄：

```text
result.width = max(main.width, 420)
```

### 7.3 跟随策略

建议第一阶段采用“打开时定位”，不实时跟随拖动：

- 用户提交命令时，读取主窗口当前位置，定位结果窗口。
- 用户拖动主窗口后，下一次打开结果窗口时再重新定位。

第二阶段再支持实时跟随：

- 监听 `main.onMoved`。
- 若结果窗口未固定，则同步移动。

这样可以避免第一阶段引入过多窗口同步复杂度。

## 8. 前端状态拆分

### 8.1 当前问题

当前 `CommandCard` 同时负责：

- 输入框。
- 窗口大小。
- 设置面板。
- AI 消息列表。
- AI 流式状态。
- 结果区域。

双窗口后，需要把“长条入口状态”和“结果内容状态”分开。

### 8.2 建议状态

主窗口保留：

```ts
interface CommandBarState {
  inputValue: string;
  status: "idle" | "generating" | "done" | "error" | "file-results";
  statusText: string;
  settingsOpen: boolean;
}
```

结果窗口使用：

```ts
interface ResultState {
  visible: boolean;
  pinned: boolean;
  mode: "chat" | "file" | "empty" | "error";
  messages: ChatMessage[];
  fileResults: FileSearchResult[];
  generating: boolean;
  error?: string;
}
```

第一阶段可以不引入复杂全局状态库。推荐用 Tauri event 在两个窗口之间同步轻量事件。

## 9. 窗口间通信方案

### 9.1 推荐方式

使用 Tauri event 作为两个 WebView 窗口之间的轻量通信通道。

主窗口发事件：

```text
result:show
result:hide
result:set-mode
result:file-results
result:chat-start
result:chat-delta
result:chat-done
result:error
```

结果窗口发事件：

```text
command:stop-chat
command:clear-context
command:close-result
command:pinned-changed
```

### 9.2 AI 流式事件归属

有两种可行方式：

#### 方案 A：主窗口继续发起 AI 请求

主窗口调用 `streamChat()`，收到 token 后：

- 更新主窗口短状态。
- 通过 Tauri event 把 delta 转发给结果窗口。

优点：

- 改动小。
- 复用当前 `CommandCard` 中的 AI 发起逻辑。

缺点：

- 主窗口仍持有聊天内容状态，职责不够干净。

#### 方案 B：结果窗口发起 AI 请求

主窗口解析命令后：

- 打开结果窗口。
- 发送 `result:chat-start` 事件和 prompt。
- 结果窗口调用 `streamChat()` 并维护消息列表。

优点：

- 职责更清晰。
- 长内容状态归结果窗口。

缺点：

- 需要确保结果窗口已经创建并完成事件监听。

### 9.3 推荐落地选择

第一阶段建议采用方案 A，降低改动风险。

第二阶段再迁移到方案 B，让结果窗口成为真正的内容容器。

## 10. 实施阶段

### 阶段 1：主窗口固定长条

目标：

- 停止把主窗口拉高为矩形。
- 去掉主窗口内嵌 `result-panel` 的主要展示职责。

改动：

1. `CommandCard.tsx` 中保留输入栏与设置面板。
2. `setExpanded(true)` 不再调用 `win.setSize(new LogicalSize(WIDTH, EXPANDED_H))`。
3. 主窗口高度固定为 `COLLAPSED_H`。
4. `window.expanded` 暂时只作为兼容字段保留，不再控制主窗口高度。
5. 输入后通过结果窗口展示内容。

验收：

- 无论输入 AI、`/file`、`?`，主窗口都不变成大矩形。
- 拖动、锁定、位置记忆仍正常。

### 阶段 2：创建结果窗口

目标：

- 新增独立结果窗口 `result`。
- 主窗口提交命令时能打开结果窗口。

改动：

1. 新增 `src/features/result/resultWindow.ts`。
2. 封装：
   - `ensureResultWindow()`
   - `showResultWindow()`
   - `hideResultWindow()`
   - `positionResultWindowNearMain()`
3. 新增 `src/components/ResultWindow.tsx`。
4. `App.tsx` 根据当前窗口 label 渲染不同根组件：

```tsx
const label = getCurrentWindow().label;

if (label === "result") {
  return <ResultWindow />;
}

return <CommandCard />;
```

验收：

- 输入 AI 后出现独立浮层。
- 浮层透明、无边框、不进任务栏。
- 浮层不会改变主窗口尺寸。

### 阶段 3：迁移聊天显示

目标：

- AI 回复在结果窗口显示。
- 主窗口只显示短状态。

改动：

1. 复用现有 `ChatView.tsx` 到 `ResultWindow.tsx`。
2. 主窗口发起 `streamChat()`。
3. 主窗口收到 delta/done/error 后，通过事件同步给结果窗口。
4. 结果窗口维护自己的 `messages` 镜像。

验收：

- AI 流式 token 在结果窗口逐步显示。
- 停止生成按钮能继续调用 `stopChat()`。
- 清空上下文按钮能调用 `clearContext()` 并清空结果窗口消息。

### 阶段 4：结果窗口关闭与固定

目标：

- 结果窗口具备关闭与固定行为。

改动：

1. 在 `ResultWindow` 顶部增加轻量工具栏：
   - 关闭。
   - 固定/取消固定。
   - 停止生成。
   - 清空上下文。
2. `Esc` 行为：
   - 未固定时隐藏结果窗口。
   - 固定时不隐藏。
3. 点击主窗口外部不强制关闭结果窗口，避免误消失。

验收：

- 关闭结果窗口不影响主窗口。
- 固定后结果窗口保持显示。
- 取消固定后 Esc 可以隐藏。

### 阶段 5：文件搜索接入

目标：

- `/file` 搜索结果进入结果窗口。

改动：

1. 新增 `src-tauri/src/file_search.rs`。
2. 注册 Tauri 命令：
   - `search_files(query)`
   - `open_file(path)`
   - `reveal_file(path)`
3. 新增前端结果组件：
   - `FileResultsView.tsx`
4. `CommandCard.tsx` 的 `/file` 分支：
   - 打开结果窗口。
   - 显示搜索中状态。
   - 调用 `search_files`。
   - 发送结果到结果窗口。

验收：

- `/file 论文` 可展示本地文件名搜索结果。
- 点击文件可打开。
- 点击所在文件夹可定位。

### 阶段 6：设置项完善

目标：

- 让用户控制结果窗口行为，但保持设置克制。

新增设置建议：

```ts
interface ResultWindowSettings {
  width: number;
  height: number;
  placement: "auto" | "bottom" | "top" | "right";
  auto_hide: boolean;
  opacity_linked: boolean;
}
```

第一阶段可以只持久化：

- 结果窗口宽度。
- 结果窗口高度。
- 是否固定。

不建议第一阶段加入过多样式项。

验收：

- 用户调整结果窗口大小后可记住。
- 主窗口设置仍保持简洁。

## 12. 视觉规范

### 12.1 主窗口

主窗口必须保持：

- 长条。
- 轻盈。
- 低信息密度。
- 不展示长文本。
- 不出现大面积矩形结果区。

推荐尺寸：

```text
默认：420 x 64
可调宽度：360 - 640
固定高度：56 - 72
```

### 12.2 结果窗口

结果窗口可以更像轻量面板，但仍应保持克制：

- 半透明玻璃背景。
- 圆角与主窗口一致。
- 宽度默认与主窗口一致，必要时最小 420。
- 高度默认 320 到 420。
- 内容区滚动，不撑大窗口。
- 工具栏紧凑，不喧宾夺主。

推荐布局：

```text
┌──────────────────────────────┐
│ AI 对话                  × 📌 │
├──────────────────────────────┤
│                              │
│ 消息 / 文件结果 / 错误状态    │
│                              │
├──────────────────────────────┤
│ 停止生成  清空上下文          │
└──────────────────────────────┘
```

## 13. 关键风险与处理

### 13.1 结果窗口事件监听时序

风险：

- 主窗口创建结果窗口后立刻发送事件，结果窗口尚未完成监听，导致首条消息丢失。

处理：

- `ensureResultWindow()` 后等待窗口发出 `result:ready`。
- 或在主窗口侧维护最后一次结果状态，结果窗口 ready 后主动拉取。

第一阶段建议：

- 结果窗口首次创建后延迟发送初始化事件。
- 后续迭代再实现 ready 握手。

### 13.2 多窗口状态重复

风险：

- 主窗口和结果窗口各自维护消息，可能不一致。

处理：

- 第一阶段接受“主窗口为源，结果窗口为镜像”。
- 第二阶段迁移为“结果窗口为源，Rust ChatState 为后端上下文源”。

### 13.3 结果窗口位置越界

风险：

- 主窗口靠近屏幕边缘时，结果浮层跑出屏幕。

处理：

- 第一阶段至少处理上下方向。
- 第二阶段使用当前 monitor work area 做完整边界修正。

### 13.4 透明窗口焦点体验

风险：

- 结果窗口出现后抢焦点，影响继续输入。

处理：

- AI 结果窗口可以抢焦点，因为用户通常要看结果。
- `/file` 结果窗口可以抢焦点，方便键盘选择。
- `/web` 不打开结果窗口，避免干扰。

### 13.5 设置面板与结果窗口冲突

风险：

- 打开设置时结果窗口仍显示，视觉混乱。

处理：

- 打开设置时，如果结果窗口未固定，则隐藏结果窗口。
- 如果结果窗口已固定，则保持显示。

## 14. 验收标准

### 14.1 主窗口美感验收

- 应用启动后只出现长条搜索框。
- 输入 AI 问题后，主窗口尺寸不变化。
- 文件搜索后，主窗口尺寸不变化。
- 浏览器搜索后，主窗口尺寸不变化。
- 主窗口没有长期占据桌面的矩形内容区。

### 14.2 结果浮层验收

- AI 回复出现在独立结果窗口。
- 流式输出正常。
- 停止生成正常。
- 清空上下文正常。
- 结果窗口可关闭。
- 结果窗口可固定。
- 未固定时 Esc 可隐藏结果窗口。

### 14.3 窗口行为验收

- 主窗口拖动和锁定仍正常。
- 主窗口位置记忆仍正常。
- 结果窗口默认显示在主窗口附近。
- 主窗口靠近屏幕底部时，结果窗口不会跑出屏幕。
- 结果窗口不进入任务栏。

### 14.4 回归验收

- 设置页仍可打开。
- 外观设置仍能影响主窗口。
- API Key 仍只存系统凭据管理器。
- 浏览器搜索仍能打开默认浏览器。
- `pnpm build` 通过。
- `cargo check` 通过。
- `pnpm tauri dev` 可启动并手动验证双窗口。

## 15. 推荐优先级

建议开发顺序：

1. 先固定主窗口为长条，不再内嵌展开结果区。
2. 再实现结果窗口的创建、显示、隐藏和定位。
3. 再把 AI ChatView 迁移到结果窗口。
4. 再补关闭、固定、Esc 行为。
5. 最后接入 `/file` 搜索结果。

这样做的好处：

- 第一阶段就能立刻解决“长条变矩形破坏美感”的核心问题。
- 每一步都有可手动验证的产物。
- 不会一次性重构 AI、搜索、设置和系统托盘。

---

## 16. 最终建议

Bugzia 的主视觉资产应该是主窗口长条，而不是结果面板。

因此推荐把产品规则固定下来：

> 主窗口永远是桌面上的漂亮搜索框；所有长内容都进入可消失、可固定的独立结果浮层。

这个规则能同时满足两个需求：

- 平时，它是桌面美化组件。
- 使用时，它是轻量 AI 与搜索工具。

后续所有功能扩展都应遵守这个边界：不要再让主窗口承担长内容展示职责。
